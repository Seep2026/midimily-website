# Midimily Web Deck Design Rules

本文档是 Midimily Web Deck 的专用设计规则。未来 Codex 新增或修改 `decks/*/slides.md` 前，必须先读取：

1. `design-system/MIDIMILY_MASTER.md`
2. `design-system/MIDIMILY_DECK.md`
3. 对应 deck 的 `BRIEF.md`

Web Deck 是方案的主体验，不是文章页，也不是传统下载型 PPT。它需要让用户用较短时间理解一个判断、一条路径或一个行动方案。

## 1. Deck 封面页规则

目标：

- 让用户 3 秒内知道这个方案讲什么。
- 建立 Midimily 的专业、温和、轻科技气质。
- 不把封面做成营销海报。

内容结构：

- 主标题：1 个清晰判断或方案名称。
- 副标题：解释范围、方法或受众。
- 可选脚注：方案类别或 Midimily Web Deck，但如果封面已很满，可以不写。

标题建议：

- 12-24 个汉字为宜。
- 最多不超过 32 个汉字。
- 避免过长并列结构。

视觉规则：

- 标题区域占页面宽度约 70%-75%。
- 主标题左上排列，封面可略微更大，但不要压满画面。
- 背景使用暖白、浅蓝灰、低饱和蓝紫、浅青。
- 可以有极轻节点、圆弧、细线，但不能抢标题。

推荐 layout：

- `cover`

不建议：

- 不使用强口号。
- 不使用“震撼发布”“终极指南”等夸张表达。
- 不加入下载按钮、导出按钮、二维码堆叠。

## 2. 观点页规则

目标：

- 每页只讲一个核心判断。
- 让用户读完后能记住一句话。

内容结构：

- 标题：一个观点。
- 副标题或短解释：补充判断。
- 可选 2-4 条要点。
- 可选底部强调句。

标题写法：

- 使用判断句，而不是名词堆叠。
- 推荐：
  - “工具尝鲜不等于业务落地”
  - “岗位不是未来才减少”
  - “Agent 的关键，是 AI 开始做决策”
- 避免：
  - “AI 工具介绍”
  - “背景说明”
  - “相关分析”

视觉规则：

- 主标题固定左上。
- 内容区可居中，但不能挤压标题。
- 如果只有一句判断，可以留大量空白。

推荐 layout：

- `statement`
- `points`，当需要少量要点时使用。

## 3. 对比页规则

目标：

- 帮用户理解“过去 vs 现在”“误区 vs 正解”“Workflow vs Agent”等差异。

内容结构：

- 标题：明确对比主题。
- 两组内容，每组 3-5 条。
- 可选底部结论。

常见对比：

- 过去的安全感 / 新的安全感。
- 工具尝鲜 / 流程落地。
- AI Workflow / AI Agent。
- 短期 ROI / 长期 ROI。
- 人设定路径 / AI 判断路径。

视觉规则：

- 两列布局优先。
- 移动端自然堆叠。
- 两列不要使用强烈红绿对立。
- 用浅蓝紫和浅青做轻区分。

推荐 layout：

- `points`
- `process`，如果对比同时带有阶段关系。

写法限制：

- 每列最多 5 条。
- 每条不超过 18 个汉字。
- 不写长段落解释。

## 4. 路径页规则

目标：

- 给出用户可理解、可执行的行动路径。
- 让方案从“观点”进入“下一步怎么做”。

内容结构：

- 标题：路径核心。
- 3-5 个步骤。
- 每个步骤包含短标题 + 一句解释。
- 可选底部行动建议。

常见路径：

- 场景梳理 → 工具选型 → 流程接入 → 培训陪跑。
- 工具掌握 → 效率提升 → 能力重塑 → 路径规划。
- 目标定义 → 步骤拆解 → 工具选择 → 结果检查 → 模板沉淀。

视觉规则：

- 使用清晰编号。
- 编号不能和文字重叠。
- 步骤卡片要有足够留白。
- 不要把 6 个以上步骤塞进一页。

推荐 layout：

- `process`
- `points`

## 5. 案例页规则

目标：

- 用具体场景帮助用户代入。
- 不写成案例长文。

内容结构：

- 标题：案例场景或入口。
- 3-4 个场景卡片。
- 每个卡片只写一个任务或变化。
- 可选一句“这说明什么”。

适合内容：

- 内容运营。
- 客服支持。
- 知识管理。
- 销售辅助。
- 个人写作、研究、整理、复盘。
- AI Coding Agent 场景。

视觉规则：

- 采用轻卡片或分栏。
- 背景不要复杂。
- 场景名称要比解释更突出。

推荐 layout：

- `points`
- `process`

写法限制：

- 不讲完整故事。
- 不堆数据。
- 不使用伪案例和夸大结果。

## 6. CTA 页规则

目标：

- 自然收束方案。
- 给用户一个低压力下一步。

内容结构：

- 标题：一个温和的行动问题。
- 副标题：说明 Midimily 可以陪用户从哪里开始。
- 1 个 CTA 按钮。
- 可选 1-3 条行动建议。

CTA 文案：

- 企业方案：
  - 预约企业 AI 咨询
  - 从第一个 AI 场景开始
  - 从第一个 OPC 试点开始
- 个体成长方案：
  - 预约个体成长咨询
  - 梳理个人 AI 工作流
  - 从一个真实任务开始

CTA 链接：

- 默认使用 `/#contact`。

视觉规则：

- 最后一页不要像硬广。
- 可以留更多空白。
- 按钮清楚，但不使用高饱和颜色。

推荐 layout：

- `cta`

## 7. 每页文字长度限制

硬性建议：

- 每页只讲一个重点。
- 每页要点不超过 5 条。
- 优先 3-4 条。
- 每条要点不超过 18 个汉字。
- 每页总字数建议控制在 120 个汉字以内。
- 内容复杂页最多不超过 180 个汉字。
- CTA 页最多不超过 120 个汉字。

如果超过限制：

- 拆成两页。
- 合并重复要点。
- 把解释改成底部强调句。
- 删除不影响理解的背景描述。

禁止：

- 不写 3 行以上连续段落。
- 不把 Markdown 原文整段搬进 slides.md。
- 不在同一页同时讲背景、问题、路径、案例、CTA。

## 8. 标题字数建议

标题是 Deck 的骨架。

建议：

- 封面标题：12-24 个汉字。
- 观点页标题：8-22 个汉字。
- 路径页标题：8-20 个汉字。
- 对比页标题：8-22 个汉字。
- CTA 页标题：8-20 个汉字。

标题风格：

- 用判断句。
- 用动词。
- 少用抽象名词。
- 不使用过度营销语。

好标题示例：

- 企业不是没有用 AI
- 工具尝鲜不等于业务落地
- 先从高频业务开始
- AI 不只是一个工具
- 下一代能力，是管理 AI 完成任务

弱标题示例：

- 背景介绍
- 方案说明
- AI 相关能力
- 项目价值分析
- 总结与展望

## 9. 中文排版规则

基础规则：

- 中文短句优先。
- 中文标点使用全角。
- 英文术语前后留空格，如 `AI Agent`、`Web Deck`、`Vibe Coding`。
- 数字与中文之间可不强制空格，但时间、比例、公式要清楚。
- 不使用过多括号。
- 不使用连续感叹号。

换行规则：

- 标题可以主动断行，但断行应符合语义。
- 不把一个词拆开换行。
- 要点不超过两行。
- CTA 按钮文字不换行。

中英文混排：

- 必要术语可以保留英文。
- 第一次出现复杂术语时，用中文解释。
- 不堆砌技术缩写。

公式规则：

- 可以出现少量公式，如 `2^4 = 16`、`2 ↑↑ 4 = 65,536`。
- 公式必须服务于观点，不做数学炫技。
- 公式页也要有一句中文解释。

## 10. 禁止项

内容禁止：

- 不制造失业恐慌。
- 不自吹自擂。
- 不空喊“赋能”“颠覆”“重塑未来”。
- 不堆砌 AI 黑话。
- 不写成长文章。
- 不加入 PDF/PPT 下载 UI。
- 不加入导出按钮。
- 不加入登录、后台、CMS 入口。

视觉禁止：

- 不使用赛博朋克。
- 不使用强霓虹。
- 不使用黑金风。
- 不使用传统咨询公司模板。
- 不使用大面积紫粉渐变。
- 不使用厚重白边和黑边播放器感。
- 不使用复杂边框套边框。
- 不使用过度动画。

布局禁止：

- 不让内容溢出 Deck 面板。
- 不让编号和文字重叠。
- 不让标题因内容多少动态跑到页面中间。
- 不把所有内容都居中堆叠。
- 不让移动端横向滚动。

## 11. Slidev Layout 使用建议

当前优先使用已有 Midimily theme layout，不轻易新增复杂 layout。

推荐对应关系：

| 内容类型 | 推荐 layout | 说明 |
| --- | --- | --- |
| 封面页 | `cover` | 标题 + 副标题 + 可选短脚注 |
| 强观点页 | `statement` | 一句话判断，留白充足 |
| 要点页 | `points` | 3-5 条要点 |
| 路径页 | `process` | 3-5 个阶段或步骤 |
| 对比页 | `points` | 用两组要点表达对比 |
| 案例页 | `points` / `process` | 场景卡片或步骤卡片 |
| CTA 页 | `cta` | 结尾行动建议 |

使用顺序建议：

1. 封面：`cover`
2. 核心判断：`statement`
3. 问题或误区：`points`
4. 结构或对比：`points`
5. 路径：`process`
6. 场景或案例：`points`
7. 行动建议：`points` / `process`
8. CTA：`cta`

新增 layout 的条件：

- 现有 layout 无法表达必要结构。
- 新 layout 能复用于至少 3 个 deck。
- 不为了单页炫技新增 layout。

## 12. 企业方案与个体成长方案差异

共同点：

- 都保持轻科技、温馨商务、清晰克制。
- 都每页只讲一个重点。
- 都需要从问题走向路径，再到行动。
- 都不制造焦虑。

企业 AI 落地方案：

- 色彩倾向：浅蓝紫、蓝灰。
- 关键词：场景、流程、工具、培训、陪跑、组织、ROI、业务闭环。
- 受众：企业家、业务负责人、团队管理者、运营/客服/知识管理团队。
- 语气：清醒、专业、可落地。
- 内容重点：
  - 业务场景是否真实。
  - 流程是否清楚。
  - 团队是否能用起来。
  - 是否能沉淀组织经验。
- CTA 倾向：
  - 预约企业 AI 咨询。
  - 从第一个 AI 场景开始。
  - 从一个小闭环开始。

个体 AI 成长方案：

- 色彩倾向：浅青蓝、暖白。
- 关键词：工具、学习、工作、求职、成长、工作流、作品集、竞争力。
- 受众：职场人、学生、程序员、产品/运营人员、AI 工具学习者。
- 语气：温和、建设性、有陪伴感。
- 内容重点：
  - 如何降低焦虑。
  - 如何从一个真实任务开始。
  - 如何形成 AI 工作流。
  - 如何沉淀模板和作品。
- CTA 倾向：
  - 预约个体成长咨询。
  - 梳理个人 AI 工作流。
  - 从一个真实任务开始。

## 13. 生成前检查清单

Codex 生成 `slides.md` 前必须确认：

- [ ] 已读取 `MIDIMILY_MASTER.md`。
- [ ] 已读取本文件。
- [ ] 已读取对应 `BRIEF.md`。
- [ ] 明确 deck 类型：enterprise 或 individual。
- [ ] 明确目标用户。
- [ ] 明确读完后希望用户做什么。
- [ ] 规划了每页唯一重点。
- [ ] 页面数量与 `meta.json.pageCount` 一致。
- [ ] 最后一页有自然 CTA。
- [ ] 没有 PDF/PPT/导出/登录入口。

## 14. 发布前检查清单

生成或修改 Deck 后必须检查：

- [ ] 标题没有过长。
- [ ] 单页没有长段落。
- [ ] 要点没有超过 5 条。
- [ ] 编号和文字不重叠。
- [ ] 内容没有溢出 Deck 面板。
- [ ] 企业/个体色彩倾向正确。
- [ ] CTA 链接指向 `/#contact`。
- [ ] `pnpm deck:build -- {slug}` 可通过。
- [ ] `pnpm build` 可通过。
